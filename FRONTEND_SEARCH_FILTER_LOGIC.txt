========================================================
  SEARCH ↔ FILTER RELATIONSHIP — FRONTEND INTEGRATION GUIDE
========================================================

CONTEXT
-------
There are TWO different places in the UI where the user can type a search:

  1. HEADER SEARCH BAR  — global search visible on every page
  2. FILTER PANEL SEARCH — a search field inside the active filter panel
                           (while the user is browsing a product list)

These two searches behave DIFFERENTLY and must be wired DIFFERENTLY.


========================================================
  1. HEADER SEARCH BAR  (global search)
========================================================

PURPOSE
  The header search is a "go to results" action.
  When the user submits a search from the header, they want to see
  ALL matching products — ignoring any category or filter they had open.

ENDPOINT TO USE
  GET /products?lang=nl&q=polo&page=1&limit=24

BEHAVIOR
  - When the user searches from the header:
    → Navigate to the product list page
    → Reset ALL active filters (colors, sizes, brands, price, category, etc.)
    → Pass only q= (and lang, page, limit)
    → Do NOT carry over filters from a previous browsing session

  - The filter panel should also reload with:
    GET /products/filters?lang=nl&q=polo
    (so the panel only shows filters that are relevant to the search results)

EXAMPLE FLOW
  User is browsing "headwear-accessories" with colors=RED active.
  → User types "polo" in the header and hits Enter.
  → Reset: categorySlug, subCategorySlug, colors, sizes, brands, price, etc.
  → Call: GET /products?lang=nl&q=polo
  → Call: GET /products/filters?lang=nl&q=polo
  → Filters panel now reflects only what exists in polo search results.


========================================================
  2. FILTER PANEL SEARCH  (search within current results)
========================================================

PURPOSE
  The filter panel search narrows down WITHIN what the filter already shows.
  It does NOT reset filters — it adds q= on top of them.

ENDPOINT TO USE
  GET /products?lang=nl&q=polo&categorySlug=headwear-accessories&colors=RED
  GET /products/filters?lang=nl&q=polo&categorySlug=headwear-accessories&colors=RED

BEHAVIOR
  - When the user types in the filter panel search field:
    → Keep ALL active filters (categorySlug, colors, sizes, brands, price, etc.)
    → Add &q=<search term> to BOTH endpoint calls
    → The API intersects the search results with the active filters

  - When the user clears the filter panel search:
    → Remove q= from both calls
    → Keep all other filters as they were

EXAMPLE FLOW
  User is on "headwear-accessories" with colors=RED active.
  → User types "cap" in the filter panel search box.
  → Both calls become:
      GET /products?lang=nl&categorySlug=headwear-accessories&colors=RED&q=cap
      GET /products/filters?lang=nl&categorySlug=headwear-accessories&colors=RED&q=cap
  → Result: only RED caps in headwear-accessories.
  → Filters panel updates to show only what is available in that intersection.


========================================================
  RULES — q PARAM BEHAVIOR
========================================================

- q is optional on both endpoints.
- Minimum 2 characters — do not call the API with q shorter than 2 chars.
- q works together with ALL other params:
    categorySlug, subCategorySlug, colors, sizes, brands,
    minPrice, maxPrice, organic, recycled, sortBy, page, limit
  All filters remain active when q is present.
- If q returns 0 matches, the API returns an empty product list
  and an empty filter panel (this is correct — show "no results" UI).


========================================================
  SUMMARY TABLE
========================================================

  Search source          | Reset filters? | q param added to?
  -----------------------|----------------|-------------------------------
  Header search bar      | YES — reset all| q only (no other filters)
  Filter panel search    | NO — keep all  | q + all existing filter params


========================================================
  BOTH ENDPOINTS NOW ACCEPT q
========================================================

  GET /products
    Params: lang, q*, categorySlug*, subCategorySlug*, colors*, sizes*,
            brands*, minPrice*, maxPrice*, organic*, recycled*,
            sortBy*, page*, limit*   (* = optional)

  GET /products/filters
    Params: same as above (q was already supported here)

  The API intersects search matches with all active filters on both endpoints.
  No separate /products/search endpoint is needed for filter-aware search.
